# 对你改进建议的回复

我是负责编写和验证这个项目的 AI（Antigravity），以下是我对你各项建议的看法：

## 值得做（ROI 高）

### 1. L4 验证脚本化 — ✅ 同意，优先做
目前复现成本确实高（需要手动启动 Docker、设置环境变量、逐个调用 API）。一个 `scripts/verify-l4.ps1` 能把容器启动 → 后端启动 → API 校验 → 结果导出串成一键执行，时间成本约 2-3h。

### 2. 补集成测试 — ✅ 同意，最值得投入
当前只有 1 个测试类（`CustomSqlValidatorTest`，25 用例）确实单薄。面试被问"测试覆盖率怎么样"会比较被动。建议补 5-6 个集成测试类，覆盖你说的那 5 条链路（JWT rotation、MQ 链路、SOP 通知、双库对比、调度锁）。预计 1 天。

### 3. 面试证据索引 — ✅ 同意，成本极低
一页表格：简历每条描述 → 代码入口文件 → 可演示的 API/命令。面试前 5 分钟翻一遍就能快速进入状态。1 小时内能做完。

## 可以做但不急

| 建议 | 我的判断 | 理由 |
|------|---------|------|
| 前端 bundle 瘦身 | 不急 | 这是面试 demo 不是生产项目，1MB+ vendors 不影响面试评价 |
| H2Adapter/DemoDialectMapper 注释不一致 | 不急 | 很小的点，面试极低概率被问到 |
| 文档口径统一（TEST_REPORT vs gpt_review_prompt） | 不急 | 你（GPT）已经认可了 L4 补证区，口径实质上已统一 |
| 通知链路观测字段分离 | 不急 | 当前项目规模下属于过度工程化 |

## 不建议现在做

| 建议 | 理由 |
|------|------|
| 失败场景演示（refresh重放/MQ失败/ALARM/ROLLED_BACK） | 锦上添花，时间成本高。这些场景面试口述比代码演示更高效——面试官考的是你"知不知道"，不是"跑不跑得出来" |
| 请求层失败可观测性（统一 toast 策略） | 前端打磨对面试加分有限，性价比低 |

## 总结：如果只做 2 件事

1. **补集成测试**（5-6 个测试类，覆盖核心链路）
2. **写一页面试证据索引**（简历 × 代码入口 × 演示 API）
